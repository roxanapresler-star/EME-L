## Логирование данных в ЕМЕ БД

### Расположение и структура логов

Директория с логами работы ЕМЕ БД находится внутри директории с данными (`dat0ws\logs`). Основной лог-файл — **CATASTRF.log**. Внутри директории logs могут находиться поддиректории с логами различных компонентов системы.

### Логи по компонентам системы EME.WMS

У каждого экземпляра системы свои логи в ЕМЕ БД:

| Компонент | Содержание логов |
|---|---|
| **Сервер** | Создание архивов для выравнивания, подключение рабочих станций, сохранение файлов транзакций |
| **Рабочая станция** | Действия пользователей и системы (залогированные программистом), информация о выравнивании |
| **HTTP-сервер** | HTTP-подключения (ТСД, веб-сервисы) + все события обычных рабочих станций |
| **Мозговой компьютер** | Регламентные задания + все события обычных рабочих станций |
| **Сервер заданий** | Получение/выполнение/ошибка заданий + все события обычных рабочих станций |
| **РС (http + сервер заданий)** | HTTP-подключения + задания + все события обычных рабочих станций |
| **Монопольный (/http + /bc)** | HTTP-подключения + регламентные задания + все действия пользователей и системы |

### Выбор логов для анализа проблемы

Для эффективного анализа логов в системе EME.WMS необходимо понимать место возникновения события:

- **Проблема с ТСД** → логи HTTP-сервера
- **Проблема с выгрузкой данных в SQL по будильнику** → логи мозгового компьютера
- **Ошибки при работе в рабочей станции** → CATASTRF.log рабочей станции

## Методика решения вопросов с производительностью EME.WMS

### Общий подход

Первым делом необходимо детально выяснить, какие действия приводят к снижению производительности EME.WMS. Каждый конкретный пример нужно разобрать, выяснить причину и только после этого принимать решение о необходимых работах.

### Типовые случаи падения производительности

1. Диспетчер терминалов долго открывается
2. Приказы долго перекидываются на ТСД через Диспетчер терминалов
3. Подвисания на ТСД, долгий отклик сервера («жёлтый глаз»)
4. Ожидание очереди транзакций на рабстанции
5. В браузерах долго применяются фильтры
6. Долго открывается конкретный справочник или диалог документов
7. Долго ставятся в буфер строки документов

**Важно:** Ускорение путём оптимизации кода — не единственный способ. Устранение ожидания отклика программы (предзаготовка данных, фоновое сохранение) также повышает производительность в системе EME.WMS.

### Обязательные мероприятия (независимо от проблемы)

1. **Ежедневная перезагрузка службы сервера EME.WMS** — чем дольше работает, тем сильнее падает производительность

2. **Проверка CATASTRF.log** каждого компонента (сервер, мозговой ПК, HTTP-сервер, рабочие станции) на наличие ошибок. Обработка ошибок сама по себе снижает производительность

3. **Проверка кэширования класса Terminals** — флаг «ВЗ» (веб-запрос) в редакторе классов должен быть включён

4. **Проверка пакетной обработки ТСД** — транзакции ТСД должны обрабатываться на сервере заданий «пачкой». Строка в исходниках ТСД не должна быть закомментирована:
   ```cpp
   csRequestString += CEMEString("&%1=%2").arg("JobType").arg("TRANSACTION:BATCH");
   ```

5. **Тест производительности сервера** — рабочая станция: «Система → Тест сети» → «Тест скорости компьютера». Современные компьютеры: ~300%. Ниже 60% — требуется обновление железа. Выше 60% — менять только если сервер — узкое место

6. **Диагностика стека потоков** на HTTP-сервере через «ЕМЕ БД сервер менеджер»

### Примеры решения проблем с производительностью

#### 1. Долго открывается Диспетчер терминалов

- **Причина:** большое количество «мусорных» приказов (битые строки записи «Очередь приказов»)
- **Решение:** удаление мусорных строк специальным доктором. Связанные строки в записи «Приказ на перемещение» не удалять, а проставить признак `SetDeleted()`. Время запуска доктора согласовать со складом. Выяснить причину появления таких приказов и устранить

#### 2. Долго отправляются приказы на ТСД

- **Причина:** очередь транзакций на сервере и неудачная доработка транзакции отправки
- **Решение:** оптимизировать транзакцию — вынести все вычисления, оставить только проверку изменения данных и их внесение в базу. Устранить причину неудачной доработки путём профилирования

#### 3. Долго открывается диалог

- **Причина:** неправильно сделанный skip-mode или интегратор с неправильной группировкой
- **Решение:** профилирование, поиск причины, оптимизация кода. Пересмотреть группировку в интеграторе

#### 4. Долгая постановка строк в буфер

- **Причина:** неудачная доработка, раздутые справочники, неудачные настройки стратегии
- **Решение:** профилирование кода, поиск узкого места. Уменьшить размер справочника / сделать статический объект Map с обновлением перед открытием транзакции. Пересмотреть галочки в стратегиях размещения (особенно связанные с поиском ячеек)

#### 5. Долгая продолжительность транзакций

- **Причина:** долгая работа сервера, неоптимальный код транзакции
- **Решение:** анализировать статистику в `REPORTS.log` («Система → Ресурсы → Отчёты о системе»). Настроить ежедневную перезагрузку сервера. Определить транзакции к оптимизации, выполнить профилирование. Цель — ускорение в 10+ раз

#### 6. Сортировка занимает много времени

- **Причина:** использование неоптимальных типов сортировки
- **Решение:** отредактировать сортировочные поля, отметить неиспользуемые. Базовая версия EME.WMS — эталонная структура. Использовать только сортировки-цепочки (для заполнения первого поля). Не использовать сортировки по реляционному пути

## Печать стека потоков на HTTP-сервере

Простой и эффективный способ обнаружения неоптимального кода HTTP-запроса на сервере EME.WMS.

### Порядок действий

1. Подключитесь к серверу WMS в момент максимальной нагрузки
2. Откройте «ЕМЕ БД сервер менеджер» (файл `service.exe`), выберите службу продуктивного сервера → «Консоль сервера»
3. Перейдите на вкладку «Пользователи», выберите HTTP-сервер (строка окрасится в синий)
4. Нажмите правую кнопку мыши → «Записать в журнал → Стек вызова»

В `CATASTRF.log` будет выведен стек каждого потока на HTTP (как C++, так и EME-L). Проблемный код занимает несколько потоков одновременно. Вызовите команду несколько раз — повторения указывают на проблемное место в коде в системе EME.WMS.
