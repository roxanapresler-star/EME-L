## Параметры запуска ЕМЕ БД — файл eme.config

Для настройки параметров запуска ЕМЕ БД создаётся файл `eme.config` в корне рабочего каталога. Формат строки конфигурации:

```
"{Название конфигурации}" = {режим запуска} "{полный путь до директории dat0ws}\eme.hdr" {дополнительные ключи запуска}
```

Пример конфигурации:

```
"dataslice_config" = /mono "C:\db_63\dat0ws\eme.hdr" /bc /http /d_dataslice
```

Для отображения диалогового окна параметров запуска при старте ЕМЕ БД добавляются строки:

```
"ShowDialog" = 1
"EditData" = 1
```

в системе EME.WMS файл eme.config позволяет быстро переключаться между различными конфигурациями запуска базы данных без редактирования ярлыков.

## Программный состав ЕМЕ БД

ЕМЕ БД состоит из четырёх основных программных компонентов:

| Компонент | Назначение |
|-----------|----------|
| **БД (База данных)** | Структурированное хранилище данных. Сама по себе не реализует работу склада и не предоставляет кнопки пользователю. |
| **Сервер приложений** | Обеспечивает работу бизнес-процессов, очередность записи данных в БД, следит за целостностью данных и предоставляет данные из БД. |
| **Рабочая станция** | Специальная программа, обеспечивающая интерфейс пользователя к БД. |
| **HTTP-сервер** | Сервер для клиентских устройств (Монитор Заданий и ТСД), добавляет цепочку физических событий бизнес-процесса в очередь транзакций. |

в EME БД каждый компонент выполняет строго определённую роль: БД хранит данные, сервер приложений управляет бизнес-логикой, рабочая станция обеспечивает пользовательский интерфейс, а HTTP-сервер обслуживает мобильные и веб-клиенты.

## Компоненты метапроекта ЕМЕ БД

Метапроект ЕМЕ БД состоит из следующих компонентов:

| Компонент | Описание |
|-----------|---------|
| **MetC-SyS** | Системные классы, реализации которых непосредственно касаются WMS: интерпретатор EME-L, сервер заданий, точки входа для классификаторов веб-объектов, методы вариантов, put/get-методы для различных типов данных. |
| **MetC-WMS** | Классы записей базы данных, i-классы, отвечающие за логику WMS: поиски ячеек, просчёт пачки и т.д. |
| **MetD-SyS** | Функционал для работы с диалогами в WMS системе. |
| **MetR-SyS** | Формирование отчётов и печатных форм в WMS системе. |

в языке EME-L метапроектные компоненты предоставляют готовые алгоритмы для складской логистики, диалогов и отчётности.

## Уровни программных кодов проекта ЕМЕ БД

Проекты ЕМЕ БД состоят из версии компилируемого кода и данных (комплект файлов БД). Компилируемый код включает:

- **DLL-файлы ядра** — директория `kernel/exe` (исходный код в `kernel/src`).
- **DLL-файлы метапроекта** — директория `metaproj/exe` (исходный код в `metaproj/src`).

Комплект файлов БД расположен в директории `dat0ws`.

**Уровни метапроекта (от низкого к высокому):**

1. **Ядро метапроекта** — `metc_sys.dll`, `metr_sys.dll`, `metd_sys.dll`, `metc_clr.dll`.
2. **Базовые классы** — общие для многих проектов одного типа (`metc_wms.dll` — проекты логистики адресного склада).
3. **Драйверы проектов** — DLL с метафункциями, начинаются с `outer`: `outerWMS.dll`, `outerCVP.dll` и др.

в EME БД каждому проекту соответствует свой DLL-драйвер, входящий в состав метапроекта, и отдельные данные БД. Ядро базы данных — единственный уровень взаимосвязанных библиотек, предоставляющий функции доступа к органам диалогов и ячейкам отчётов.

## Записи и поля в ЕМЕ БД — структура таблиц

ЕМЕ-БД использует обычные файлы операционной системы. База данных состоит из «прямоугольных» таблиц данных, которые называются **записи**. Строки таблиц — «строки записей», колонки — «поля записей». В полях содержатся однородные данные, поэтому у поля всегда зафиксирован атрибут типа данных.

### Доступные типы данных полей в ЕМЕ БД

| Тип данных | Описание |
|-----------|---------|
| Целое число | 2 или 4 байта |
| Число с плавающей точкой | Вещественное число |
| Строка текста | Максимальная длина задаётся размером поля |
| Массив | Массив одинаковых переменных в одном поле |
| Деньги | Финансовый тип данных |
| Ссылка | Прямая ссылка на другую запись |
| Дата | Календарная дата |
| Время | Время суток |

Поле может быть как постоянной, так и переменной длины. Для полей переменной длины создаётся дополнительное поле **VLP-указатель**. В одном поле может размещаться целый массив одинаковых переменных.

в языке EME-L таблицы называются «записи», а столбцы — «поля». Запись, содержащая свойства всех записей БД, называется «Записи БД» (аналогично «Поля БД»). Таким образом, структура БД записана в самой БД — это самореферентная архитектура.

## Данные EME.WMS на диске — файловая структура

В файлах данные размещаются по полям: одно поле записи — всегда один файл на диске. Это обеспечивает эффективность: в промышленных базах данных в одной записи бывают сотни полей, а в запросе обрабатывается лишь малое их число. При переборе и поиске данных чтение блока в несколько килобайт позволяет перебрать тысячи строк.

### Формат имён файлов данных

Файлы в директории `dat0ws` имеют имена вида `db001000.eme`, `db001001.eme`:

- `db` — признак файла данных (одинаков для всех проектов ЕМЕ-БД).
- Первые три цифры — номер записи (в примере: запись 1).
- Вторые три цифры — номер поля (поле 0, поле 1).

### Служебные файлы

| Файл | Назначение |
|------|-----------|
| `EME.HDR` | Заголовок базы данных («хедер») |
| `.INI`-файлы | Настройки пользователя; удаление всех .ini сбрасывает настройки |

в EME БД система поддерживает все глобальные файлы на всех компьютерах полностью идентичными. Процедура поддержки идентичности называется «выравнивание». Каждый экземпляр системы имеет свою копию БД в папке `dat0ws` — это необходимо для хранения обязательных данных (например, настроек подключения к серверу). Подпапки `dat0ws` не относятся напрямую к файлам БД и при копировании базы могут быть опущены.

## Выравнивание ЕМЕ БД с сервером

Сервер для выравнивания определяется IP-адресом и портом. Процедура выравнивания:

1. В рабочей станции: **Утилиты БД → Заливка проекта → Выравнивание изменений**.
2. Нажать **«Подключиться к серверу»**.
3. Нажать **«Вернуться в моно»**.
4. Должно появиться сообщение: «Выполнена базовая выгрузка проекта».

**Важные замечания:**

- БД должна быть запущена с ключом `/nodllalign` во избежание ошибок при выравнивании по модулям.
- Если выравнивается локальная БД (моно-режим) — все изменения будут потеряны.
- Если БД запущена в режиме сервера — выравнивание не требуется.

### Альтернативные способы получения данных с сервера

- При наличии `eme.config` — выбрать конфигурацию EME.WMS и нажать **«Добавить»**, ввести IP-адрес и порт сервера.
- Открыть ярлык `db.exe`.

### Внутренние версии EME — IP и порты

| Версия | IP-адрес | Порт |
|--------|----------|------|
| Релиз-версия | 10.1.15.20 | 9949 |
| Пре-прод | 10.1.15.20 | 9939 |
| Разработческая (63-й филиал) | 10.1.6.126 | 4546 |

в системе EME.WMS HTTP-сервер не отвечает за обновление программы. Для обновления используется сервер, указанный в настройках рабочей станции: **Сеть → IP-адрес сервера** и **Сеть → Порт прослушивания**.

## Когда следует делать выравнивание в ЕМЕ БД

Выравнивание рабочей станции необходимо в следующих случаях:

- **Данные повреждены** — система распознаёт повреждения по контрольным суммам (в каждом поле-файле, в каждой записи, внутри файлов — контрольные суммы сегментов).
- **Состояние рабочей станции неактуально** — вследствие отсутствия необходимых транзакций.

в EME БД данные постоянно копируются (выравниваются) по транзакциям или по файлам с сервера на каждую рабочую станцию. Это удобно для встроенной БД, поскольку транзакции могут быть сложными и продолжительными. Выравнивание позволяет выполнять все операции чтения без обращения к серверу. Также ЕМЕ БД может работать без выравнивания всей БД — в режиме NetVM, когда рабочие станции запрашивают данные с сервера по сети.

## Транзакции в ЕМЕ БД — теория и практика

### Теория транзакций

Транзакция — неразделимая группа операций, которая переводит БД из одного устойчивого состояния в другое. Устойчивое состояние означает как физическую, так и логическую целостность данных. Например, при переводе накладной в статус «Товар отгружен» все операции отгрузки по строкам документа обязаны быть объединены в одну транзакцию — иначе возникнет расхождение в остатках товаров.

В ЕМЕ-БД транзакция представляет собой буфер данных (блок памяти или файл на диске) — массив команд по записи в файлы БД новых значений («файл намерений»). Буфер транзакции сохраняется на диск для обеспечения целостности при сбоях.

в языке EME-L доступ к строкам записей осуществляется непосредственно по физическому номеру. Все ссылки — прямые по физическому номеру строки. Транзакции последовательны: нельзя открыть новую транзакцию, не развернув все предыдущие. Открытая транзакция создаёт виртуальную копию БД — «уровень данных». Уровни обеспечивают многопоточную работу.

### Открытие и закрытие транзакций

```EME-L
' Открытие транзакции '
is_transaction(1, "Моя транзакция");
' Запись в базу данных '
GTDType.Put(0);
' Закрытие транзакции '
is_transaction(-1);
```

### Вложенные транзакции

Вложенные транзакции обрабатываются как единая транзакция. Счётчик вложенности увеличивается/уменьшается:

```EME-L
' Открытие внешней транзакции '
is_transaction(1, "Моя транзакция");
' Увеличение счётчика вложенных транзакций '
is_transaction(1, tr("Вложенная транзакция"));
' Запись в базу данных '
GTDType.Put(0);
' Уменьшение счётчика вложенных транзакций '
is_transaction(-1);
' Закрытие транзакции '
is_transaction(-1);
```

### Отмена транзакций

Функция `is_abort_transaction()` помечает текущую транзакцию как «сломанную», но не закрывает её. После вызова транзакция остаётся открытой на рабочей станции, но очередь транзакций на сервере больше не удерживается. Данные не будут сохранены, но программа продолжает работу.

Вызов `is_transaction(0)` эквивалентен двум операциям:

```EME-L
is_abort_transaction();
is_transaction(-1);
```

Пример отмены вложенной транзакции:

```EME-L
' Открытие транзакции '
is_transaction(1, "Моя транзакция");

' Вложенная транзакция '
is_transaction(1, tr("Вложенная транзакция"));

' Запись в базу данных '
GTDType.Put(0);

' Отмена вложенной транзакции (помечает как сломанную, уменьшает счётчик) '
is_transaction(0);

' Закрытие транзакции '
is_transaction(-1);
```

### Исключения в транзакциях

При необработанном исключении в прикладном коде стандартный метапроектный обработчик помечает текущую транзакцию как сломанную и закрывает все вложенные транзакции.

в языке EME-L система самодиагностики основана на исключениях. **Никогда не перехватывайте исключения в коде, который пишет в базу** — это приведёт к нарушению целостности БД. Старайтесь никогда не ловить исключения самостоятельно.

## Системные записи в ЕМЕ БД

Системные записи определяют фундамент базы данных, без которого она не может запуститься. Записи имеют фиксированные номера:

| Номер записи | Назначение |
|-------------|-----------|
| −1 | Описание записей |
| −2 | Описание полей |
| −3 | Перечень возможных атрибутов полей |

Системные записи создают только программисты ЕМЕ БД.

### Обязательные записи для запуска

| Запись | Тип | Описание |
|--------|-----|----------|
| Настройки модулей | Глобальная | Изменения воздействуют на все рабочие станции и сервер |
| Настройки рабочих станций | Локальная | Управляет работой одного компьютера |
| Классы ЕМЕли | — | Хранение классов языка EME-L |
| Запросы SQL | — | Хранение SQL-запросов |
| Прикладные диалоги | — | Обязательны для работы |
| Отчёты | — | Обязательны для работы |

в EME БД все системные записи помечены специальным флагом в структуре базы данных.

## Системные поля в ЕМЕ БД

Системные поля создаются ядром БД автоматически:

| Поле | Размер | Назначение |
|------|--------|-----------|
| **Поле ключей** | 4 байта | Физический ключ строки. Уникален в масштабах всего проекта (включая все филиалы). Формируется как порядковый номер создания строки на филиале с маскированием номера филиала в старших битах. |
| **Поле флагов** | 4 байта | Битовые признаки строки: удалённость, новая строка, модифицированность (требует экспорта), факт экспорта и др. Прикладной программист редко работает с полем флагов напрямую — ядро обрабатывает его автоматически. |
| **Поле счётчиков ссылок** | 4 байта | Позволяет без перебора других записей определить наличие ссылок на данную строку. Используется для определения возможности удаления и запрета модификации. Ядро поддерживает счётчики самостоятельно. |

в системе EME.WMS прямые ссылки по физическому номеру строки в сочетании с полем счётчиков ссылок обеспечивают мгновенную проверку ссылочной целостности без необходимости сканирования таблиц.

## Чтение и запись данных в ЕМЕ БД — put_data и get_data

ЕМЕ БД относится к системам класса **RISC** (сокращённый набор команд). Используются две базовые команды:

- `put_data()` — запись данных в БД.
- `get_data()` — чтение данных из БД.

### Параметры обеих команд

| Параметр | Описание |
|----------|----------|
| Номер Записи | Целое число — адрес записи |
| Номер Поля | Целое число — адрес поля |
| Физический номер строки | Целое число — адрес строки |
| Адрес блока данных в памяти | Указатель на буфер |
| Размер блока | Размер данных |

в языке EME-L при `put_data()` адрес содержит готовый блок данных для загрузки в БД, а при `get_data()` — ссылку на свободный буфер для выгрузки. Запись и чтение выполняются очень быстро и напрямую, без операций поиска и сравнения ключей.

## Режимы запуска ЕМЕ БД

Режимы запуска определяют основной функционал базы данных:

| Режим | Описание |
|-------|----------|
| `/server` | Запуск в режиме сервера |
| `/ws` | Запуск в режиме рабочей станции |
| `/mono` | Однопользовательский режим, работа с локальной копией БД в директории `dat0ws` |

Можно выставить только один режим запуска.

### Особенности моно-режима

1. Монопольный режим возможен только на полной копии БД (невозможен на копии, запущенной ранее в режиме netvm).
2. Если экземпляр, запущенный как рабочая станция, перезапустить в моно-режиме — все изменения попадут только в локальную копию и никогда не попадут на сервер.
3. Если экземпляр из моно-режима запустить как рабочую станцию — локальная копия заменится копией сервера (произойдёт выравнивание).

## Дополнительные ключи запуска ЕМЕ БД

Дополнительные ключи определяют расширенные параметры программы:

| Ключ | Назначение |
|------|-----------|
| `/http` | HTTP-сервер для взаимодействия с Монитором Заданий и ТСД |
| `/bc` | Выполнение регламентных заданий (будильники). Пример: выгрузка данных в SQL-базу каждые 5 минут |
| `/fullvm` | Чтение данных исключительно из локального хранилища |
| `/netvm` | Взаимодействие с данными сервера WMS по сети |
| `/rundbtest` | Полный тест целостности БД |
| `/debug` | Запись отладочной информации в логи; поиск органов, модифицирующих диалог |
| `/d_emel` | Отладка EME-L кода |
| `/nodllalign` | Выравнивание без модулей (рекомендуется всегда) |
| `/d_skipdbtst` | Отключение теста целостности |
| `/d_dataslice` | Загрузка срезов данных |
| `/skiphwtest` | Отключение проверки оборудования |
| `/si` | Отключение будильников (только для мозгового компьютера) |
| `/dblevel=[номер]` | Запуск с данными на заданном номере транзакции |
| `/d_replica` | Репликация данных в SQL-базу |
| `/d_random` | Защита целостности структуры БД при стресс-тестах в «Интерфейсе ERP-WMS» |

в системе EME.WMS можно указывать неограниченное количество ключей запуска, но некоторые конфликтуют друг с другом (например: `/fullvm` и `/netvm`).

## Логирование данных в ЕМЕ БД

Директория с логами: `dat0ws\logs`. Основной лог-файл — **CATASTRF.log**. Внутри директории также могут находиться логи ТСД, сервера заданий, мозгового компьютера и др.

### Логи по экземплярам системы

| Экземпляр | Содержание логов |
|-----------|-----------------|
| **Сервер** | Создание архива для выравнивания, подключение рабочих станций, сохранение файлов транзакций |
| **Рабочая станция** | Действия пользователей и системы (предусмотренные программистом), информация о выравнивании |
| **HTTP-сервер** | HTTP-подключения (ТСД, веб-сервисы), все события рабочей станции |
| **Мозговой компьютер** | Регламентные задания, все события рабочей станции |
| **Сервер заданий** | Получение/выполнение/ошибка заданий, все события рабочей станции |
| **РС + HTTP + сервер заданий** | HTTP-подключения, задания, все события рабочей станции |
| **Моно + /http + /bc** | HTTP-подключения, регламентные задания, все действия пользователей и системы |

### Практические рекомендации по анализу логов

| Проблема | Где смотреть |
|----------|-------------|
| Проблема с ТСД | Логи HTTP-сервера |
| Проблема с выгрузкой в SQL по будильнику | Логи мозгового компьютера |
| Ошибки в рабочей станции | `CATASTRF.log` |

в EME БД для эффективного анализа логов необходимо понимать не только время события, но и место его возникновения — каждый экземпляр системы ведёт свои логи с различным набором событий.
