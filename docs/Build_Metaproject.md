## Программирование в C++ — Фабрика классов

### Назначение фабрики классов в системе EME.WMS

Таблицы БД в EME.WMS называются **записями**. Записи и поля идентифицируются именами и номерами. Строки идентифицируются индивидуальными номерами, а не по условию, как в обычных реляционных БД.

Основные константы в языке EME-L:

- `NULL_REF` — неверный или отсутствующий номер строки
- `NULL_RECORD` — неверный или отсутствующий номер записи
- `NULL_FIELD` — неверный или отсутствующий номер поля

**Важно:** Не используйте символы подчёркивания в именах органов управления на диалоге. Это конфликтует с обработчиками событий, где символ подчёркивания разделяет имя объекта и событие. Тип данных для органа управления «Список» всегда должен быть целым числом (индекс выбранного элемента).

Фабрика классов используется для реализации различных механизмов доступа к записям ЕМЕ БД. Классы программируются на C++ и являются частью таких компонентов метапроекта как **MetC_WMS** и **MetC_Sys**. Фабрика классов объявлена в заголовочном файле `innerREC/p_wms.h` (или `innerREC/p_System.h`). Чтобы использовать её функционал, необходимо прописать названия данных файлов в директиве `include`.

### Создание C++ класса для взаимодействия с записями ЕМЕ БД

Для формирования C++ класса, взаимодействующего с записями в ЕМЕ БД, выполните три шага:

1. Объявить класс в заголовочном файле (`.h`)
2. Реализовать класс в `.cpp`-файле, используя функционал фабрики классов
3. Внести класс в фабрику

#### Объявление C++ класса в заголовочном файле

В Visual Studio добавьте новый класс в существующий компонент метапроекта (например, MetC_WMS). Для объявления класса используется макрос `BEGIN_DECLARE_INNER(ИмяКласса)` как заголовок конструкции и `END_DECLARE_INNER(ИмяКласса, System)` как закрывающая часть.

Пример объявления класса в языке EME-L:

```cpp
BEGIN_DECLARE_INNER(CRecDepartament)
// Объявление членов и методов класса
END_DECLARE_INNER(CRecDepartament, System)
```

Внутри данного блока макросов выполняется перечисление используемых полей записи, инициализация полей записи, определение вызываемых из EME-L функций и определение недоступных в EME-L функций.

**Перечисление полей записи** — макросы `BEGIN_ENUM` и `END_ENUM`. Для простых полей формируется одна константа. Для полей-ссылок — две константы (первая для самой ссылки, вторая для объекта по ссылке).

```cpp
BEGIN_ENUM
Employees, EmployeesRef, Name, VacanciesCnt
END_ENUM
```

**Инициализация полей записи** — макросы `DECLARE_FLD` (простое поле) и `DECLARE_REF` (ссылочное поле).

```cpp
DECLARE_FLD(CString, Name) // тип данных C++ и название поля
DECLARE_REF(CRecEmployees, Employees) // класс для ссылочной записи и название поля
```

**Функции, доступные для EME-L**, инициализируются макросом `DECLARE_FN` с указанием количества аргументов. При повторном использовании функции фабрика определяет одни и те же символы, поэтому используются литеры для уникальности при перегрузке методов.

```cpp
// Возвращаемый тип, имя функции, литера, типы и имена аргументов
DECLARE_FN0(bool, OnComplete, A)
DECLARE_FN2(CString, CreateNewTTNNumber, A, CString, csTypeTTN, CString, csPrefix)
```

Количество аргументов может доходить до 8. Литера указывается без кавычек. Функции, не предназначенные для EME-L, объявляются по обычным правилам C++.

#### Реализация C++ класса в .cpp-файле

В `.cpp`-файле прописывается блок макросов:

```cpp
IMPLEMENT_INNER_CREATOR(CRecDepartament, System)
BEGIN_IMPLEMENT_INNER(CRecDepartament)
    // Имплементация членов и методов класса
END_IMPLEMENT_INNER(CRecDepartament)
```

Для каждого поля записи прописывается макрос `IMPLEMENT_FLD` с указанием константы из заголовочного файла:

```cpp
IMPLEMENT_FLD(Employees, "Сотрудники", "Краткое описание")
// первый аргумент — константа из перечисления
// второй аргумент — название поля в ЕМЕ БД
```

Для каждой EME-L функции прописывается макрос `IMPLEMENT_FNC`:

```cpp
IMPLEMENT_FNC(CreateNewTTNNumber, A, "Описание функции для справочной системы редактора EME-L")
```

Реализация функций с отдельным блоком макросов, где в суффиксе указывается количество аргументов:

```cpp
BEGIN_IMPLEMENT_FN1(INNER(MyClass), long, FindUserBySurname, A, CString Surname)
    // Реализация
END_IMPLEMENT_FNC
```

В конце реализации: `END_IMPLEMENT_FNC` (функция возвращает значение) или `END_IMPLEMENT_VFNC` (функция не возвращает значений).

**Важно:** В коде C++ имя класса должно использоваться с обёрткой `INNER()`. Для объявления объектов класса (не указателей) используется макрос `PROXY()`. Реализация недоступных в EME-L функций-членов описывается с `INNER()` в сигнатуре:

```cpp
INNER(Configurator)::ImportTree(char * table)
```

Добавьте макрос `REALIZE_FLD()` для обращения к переменным класса.

После добавления новых классов и полей нужно проверить их на наличие ошибок. Соберите метапроект и откройте диалог «Утилиты БД → Система → База ошибок».

#### Внесение созданного класса в фабрику

Чтобы сделать C++ класс доступным в EME-L, занесите информацию о нём в `innerREC/p_WMS.h` или `innerREC/p_System.h`.

1. Найдите конструкцию `BEGIN_DECLARE_PRJ(INNER_SYSTEM_PROJECT, CEMEPrj)` и добавьте константу имени класса в перечисление `BEGIN_REC_ENUM`.

2. Добавьте класс макросом `DECLARE_REC(ИмяКласса)`.

3. В файлах `innerREC/p_WMS.cpp` или `innerREC/p_System.cpp` добавьте макрос:

```cpp
IMPLEMENT_REC(ИмяКласса, "Имя записи в ЕМЕ БД", "Описание класса для справочной системы EME-L")
```

## Сборка метапроекта в Windows

Метапроект представляет собой расширение программ на внутреннем языке EME-L на C++ и обеспечивает работу одной версии с несколькими проектами. Каждому проекту соответствует свой dll-драйвер, входящий в состав метапроекта, и данные БД.

### Подготовка директории для сборки метапроекта в Windows

В системе Windows подготовьте директорию для сборки (например, `EMEdb`), содержащую обязательные поддиректории:

- `kernel.x64` — директория с файлами ядра
- `metaproj` — директория с файлами метапроекта

Исходный код метапроекта (базовая версия ЕМЕ БД): `http://svn.eme.ru/metaproj/trunk/metaproj/`

Библиотеки Qt и Boost: `http://svn.eme.ru/metaproj/trunk/qt/` и `http://svn.eme.ru/metaproj/trunk/boost/`

Исходный код ядра: `http://svn.eme.ru/metaproj/kernel/kernel.x64-51.1/`

### Пошаговая инструкция сборки метапроекта в Windows

1. Выгрузить из SVN исходный код библиотек Qt и Boost с помощью TortoiseSVN
2. Задать переменные среды Windows с указанием каталогов библиотек Qt и Boost
3. Выгрузить исходный код метапроекта из SVN (7 директорий: MetC_Sys, MetC_WMS, MetD_Sys, MetR_Sys, _Lib_x64, _Obj_x64, outerWMS)
4. Выгрузить исходный код ядра из SVN (директории src и exe)
5. Установить Visual Studio 2019 Professional
6. В директории outerWMS открыть файл `outerWMS.sln`
7. Настроить параметры отладки outerWMS и остальных проектов
8. Назначить outerWMS автозагружаемым проектом
9. Запустить сборку с конфигурацией **Release x64** (10–15 минут)

#### Переменные среды для библиотеки Qt

| Переменная | Значение |
|---|---|
| QMAKESPEC | `\qt\qt-4.8.6-x64-msvc2010\mkspecs` |
| QT_INCLUDE | `\qt\qt-4.8.6-x64-msvc2010\include` |
| QT_INCLUDE_V142 | `\qt\qt-everywhere-src-5.15.2\qtbase\include` |
| QT_LIB | `\qt\qt-4.8.6-x64-msvc2010\lib` |
| QT_LIB_V142 | `\qt\qt-everywhere-src-5.15.2\qtbase\lib` |
| QT_LIB_X64 | `\qt\qt-4.8.6-x64-msvc2010\lib` |
| QT5_CMAKE_DIR | `\qt\5.13.2\android_x86\lib\cmake` |
| QT5_DIR | `\qt\5.13.2` |
| QT5CORE_DIR | `\qt\5.13.2\android_armv7\lib\cmake\Qt5Core` |

#### Переменные среды для библиотеки Boost

| Переменная | Значение |
|---|---|
| BOOST_DIR | `\boost\boost_1_53_0` |
| BOOST_DIR_V142 | `\boost\boost_1_75_0` |

#### Параметры отладки метапроекта в Visual Studio

Для каждого проекта задайте:

- **Команда:** `{Рабочий каталог EME БД}\kernel.x64\exe\debug\db.exe`
- **Аргументы команды:** `/mono "{Рабочий каталог EME БД}\dat0ws/eme.hdr" /d_emel /d_dataslice`
- **Рабочий каталог:** `..\..\exe\Debug`

#### Решение проблем при сборке в Visual Studio

1. **Отсутствие заголовочных файлов** — установите пакет MFC для C++ через Visual Studio Installer (компонент «MFC C++ для средств сборки версии 142 (x64 и x86)»).

2. **Предупреждения «svn не является внутренней командой»** — установите инструменты командной строки TortoiseSVN (Modify → command line client tools → установить на локальный диск).

3. **Проект MetC_CLR не найден** — можно удалить из решения, он нужен только для сборки 3PL-портала.

## Сборка метапроекта на Linux

Метапроект можно собрать и в системе Linux. В данной инструкции используется виртуальная машина **Astra Linux** на Hyper-V. Подготовьте директорию `/opt/eme`, содержащую:

- `project_name` — файл конфигурации, бинарные файлы, скрипты разворачивания, директории данных для служб
- `azure` — склонированный репозиторий для развертывания и обновления системы

### Пошаговая инструкция сборки метапроекта на Linux

1. В Windows подготовить данные: `dat_master` (мастер-сервер) и `dat_server` (сервер раздела), задав номера разделов БД (0 и 1)
2. Подготовить систему Astra Linux (установить библиотеки, настроить кодировку)
3. Подготовить директорию `/opt/eme`
4. Отредактировать `config.json` в `/opt/eme/alm`
5. Сгенерировать службы скриптом `alm/generate_services.sh` и настроить 4 службы в моно-режиме
6. Стартовый запуск всех 4 служб в сетевом режиме
7. Подключиться к мастер-серверу и включить событие системного импорта
8. Подключиться к серверу раздела и включить события интеграционного импорта/экспорта
9. Запустить `alm/deploy.sh` для сборки метапроекта

#### Подготовка системы Astra Linux

```bash
sudo apt install libqt5gui5 libqt5core5a
sudo apt install libmicrohttpd12
sudo apt install unixodbc
```

Раскомментировать в `/etc/locale.gen`:
```
ru_RU.CP1251 CP1251
ru_RU.UTF-8 UTF-8
```
Затем выполнить `sudo locale-gen`.

#### Структура директории проекта

В `/opt/eme/project_name`:
- Скопировать директорию `alm` из репозитория
- Скопировать `dat_master` и `dat_server` (продублировать `dat_server` в `dat_server_bc` и `dat_server_http`)
- Распаковать `bin.zip` в `bin`

Символические ссылки внутри `bin`:

```bash
sudo ln -s libdb.so canal.dll
sudo ln -s libmetc_sys.so libouterwms.dll
sudo ln -s libmetc_sys.so MetC_Sys.dll
sudo ln -s libmetc_sys.so metc_sys.dll
```

#### Ключевые параметры config.json

- `project_name` — название проекта (цифры и английские буквы)
- `deploy_dir` — полный путь к директории проекта
- `bin_dir` — полный путь к директории bin
- `services` — настройки служб (ключ — название службы)
- `launch_sequence` — порядок запуска с паузами (секунды)
- `backup_dir` — путь для резервных архивов
- `system_import_service` — название службы системного импорта

#### Настройки модулей для всех служб (в системе EME.WMS на Linux)

```
Linux.Система =
Linux.Система.Драйвер БД = libmetc_sys.so;libmetc_wms.so
Linux.Система.Идентификатор БД = SYS;WMS
```

Рекомендуемые сетевые порты:
- Мастер-сервер: **3005**
- Сервер разделов: **3006**
- HTTP-сервер: **3007**

#### Проверка корректности работы служб

Для каждой службы проверьте `CATASTRF.log` в директории данных:

- **Мастер-сервер:** «Сервер транзакций загружен. Интерфейс {ip:port}»
- **Сервер раздела:** «Открыта транзакция — Выравнивание мастер-транзакций» (без ошибок между сообщениями)
- **HTTP-сервер:** «HTTP: web server started on port (XXXX)-(XXXX+1)». Тест: `curl http://localhost:3007`
- **Мозговой компьютер:** «Мозговой компьютер EME загружен.» и сообщения «Обработка событий»
