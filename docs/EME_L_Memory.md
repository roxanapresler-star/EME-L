## Работа с памятью в EME-L

В языке EME-L нет операторов `new` и `delete` — память выделяется и освобождается автоматически. Однако имеются нюансы работы с памятью, которые следует учитывать при написании программ.

Для объектов, созданных конструкцией `Object("ClassName")`, память выделяется при каждом выполнении этой конструкции.

в языке EME-L объекты, не являющиеся членами класса, удаляются автоматически при выходе из метода, в котором созданы. Поэтому утечки памяти не будет, даже если в цикле созданы тысячи EME-L объектов. Но сам код будет работать медленно.

### Правила работы с Object()

Если объект является членом класса, он создаётся при каждом выполнении `Object("ClassName")`. В таком случае есть риск утечки памяти — объект EME-L класса может существовать всё время работы сервера, а инициализировать член класса вне конструктора можно неограниченное количество раз.

Чтобы избежать медленной работы программы и утечки памяти, нужно соблюдать правила:

1. **Никогда не использовать `Object("ClassName")` внутри цикла**, даже как параметр метода.

2. **Никогда не использовать простые имена для переменных-членов класса.** Всегда добавлять префикс `m_`. Никогда не использовать префикс `m_` для локальных переменных.

3. **Стараться не инициализировать переменные-члены класса типа `Object` вне конструктора.**

4. **Использовать оператор `Const`** для создания объектов-членов класса, не имеющих глобального имени переменных и видимых только в своём методе.

в системе EME.WMS соблюдение этих правил критично для стабильной работы сервера, который может работать без перезагрузки сутками.

## Оператор Const

Оператор `Const` возвращает результат выражения, но вычисление производится один раз за всё время работы программы.

```EME-L
'Const вычисляет выражение один раз:'
NeedData = Const(is_edit_data("Report.Date"));
```

Оператор `Const` необходим в местах, где выполняется код, всегда возвращающий один и тот же результат. Если при выполнении запроса к записи с 1000 строк его не использовать — из органа диалога 1000 раз будет получено одно и то же значение.

в языке EME-L оператор Const является основным инструментом предотвращения утечек памяти и оптимизации производительности.

### Const для объектов-членов класса

В EME-L классах оператор `Const` можно использовать для создания объектов-членов класса, не имеющих глобального имени:

```EME-L
'Объект создаётся и инициализируется при первом вызове метода.'
'При последующих вызовах — обращение к созданному объекту.'
GetDocumentByRegNo(DocNoIncom)
{
    r_Doc As "Document" = Const(Object("dsDB", "Document"));
    If (~r_Doc.IsValidLine() | r_Doc.GetRegNo() != DocNoIncom)
        r_Doc.SetSkipMode();
        r_Doc.GetRegNoFld().MustBeEQ(DocNoIncom);
        r_Doc.MustBeValid();
        r_Doc.SetFirstLine();
    End If
    Return r_Doc.GetLine();
}
```

Поскольку переменная `r_Doc` не объявлена в конструкторе класса и является локальной, её нельзя изменить в других методах класса. Но благодаря `Const` она создаётся только один раз.

## Примеры утечек памяти в EME-L

### Утечка при создании Object в Map

```EME-L
'Плохой код — объект "Array" создаётся в цикле:'
For (r_Lots.SetFirstLine(); r_Lots.IsValidLine(); r_Lots.SetNextLine())
    array = mapLotRef.Lookup(line, Object("Array"));
    array.Add(r_Lots.GetLine());
    mapLotRef.SetAt(line, array);
End For
```

В цикле создаётся столько объектов "Array", сколько партий перебирается. Но используется только один, поскольку ключ "Map" не изменяется.

**Решение:** не передавать `Object("Array")` как второй параметр в метод `Lookup` у "Map" (этот параметр задаёт значение по умолчанию, если ключ не найден).

### Утечка памяти в SQL-запросах

В SQL-запросах память освобождается не так, как в EME-L классах. Сборщик мусора работает один раз после выполнения всего запроса.

```EME-SQL
'Плохой код — объект создаётся для каждой строки запроса:'
{
    dbRegs = Object("dsDB","Registers");
    dbRegs.SetLine(is_query(,"@"));
    Return dbRegs.GetQtyN() + dbRegs.GetQtyS() + dbRegs.GetQtyBuf();
}:FLOAT AS [Кол-во],
```

Это приводит к медленной работе запроса или вылету программы из-за нехватки памяти.

**Решение:** использовать оператор `Const`:

```EME-SQL
dbRegs = Const(Object("dsDB","Registers"));
```

в EME БД при работе с SQL-запросами необходимо всегда использовать Const для объектов записей, чтобы избежать накопления объектов в памяти до завершения запроса.

### Const для членов класса без глобального имени

```EME-L
'Использование Const для создания объекта-члена класса:'
DialogClassObject = Const(Object("Dialog").GetClassObject());
```

```EME-L
'Параметр записи — Const-объект:'
Parameters = Const(Object("Parameters"));
```

## Оптимизация производительности EME.WMS

Ускорение работы программы путём оптимизации кода — не единственный способ повышения производительности. Также важно устранять ожидание отклика программы: готовить пользовательские данные заранее, переносить сохранение в фоновый режим.

### Обязательные мероприятия

1. **Настроить ежедневную перезагрузку** службы сервера EME.WMS. Чем дольше сервер работает, тем сильнее падает производительность.

2. **Проверить CATASTRF.log** каждого компонента EME.WMS (сервер, мозговой ПК, HTTP-сервер, рабочие станции) на наличие ошибок. Обработка ошибок снижает производительность.

3. **Проверить кэширование** класса `Terminals` — должен быть установлен флаг "ВЗ" (веб-запрос) в редакторе классов.

4. **Обеспечить обработку транзакций ТСД на сервере заданий** с пакетной обработкой. В исходниках ТСД должна быть незакомментированная строка:
```
csRequestString += CEMEString("&%1=%2").arg("JobType").arg("TRANSACTION:BATCH");
```

5. **Выполнить тест производительности сервера** через "Система -> Тест сети" -> "Тест скорости компьютера". Современные компьютеры показывают 300% от рекомендуемой. Ниже 60% — требуется обновление железа.

6. **Провести диагностику стека потоков** на HTTP-сервере через "ЕМЕ БД сервер менеджер".

в системе EME.WMS регулярное выполнение этих мероприятий является обязательным для поддержания производительности на продуктивных складах.

### Примеры решения проблем с производительностью

**1. Долго открывается Диспетчер терминалов**

Причина: большое количество "мусорных" приказов в записи "Очередь приказов".

Решение: удаление мусорных строк специальным доктором. Связанные строки в "Приказ на перемещение" не удалять, а проставить `SetDeleted()`. Выяснить причину появления.

**2. Долго отправляются приказы на ТСД**

Причина: очередь транзакций и неудачная доработка.

Решение: оптимизировать транзакцию — вынести все вычисления, оставить только проверку изменений и запись в БД.

**3. Долго открывается диалог**

Причина: неправильно сделанный skip-mode или интегратор с неправильной группировкой.

Решение: профилирование, оптимизация кода, пересмотр группировки.

**4. Долгая постановка строк в буфер**

Причина: неудачная доработка, раздутые справочники или неудачные настройки стратегии.

Решение: профилирование, уменьшение справочника, использование статического объекта Map с обновлением перед транзакцией.

**5. Долгие транзакции**

Решение: проанализировать REPORTS.log ("Система -> Ресурсы -> Отчёты о системе"). Выполнить профилирование. Ускорение в 10 раз и более считается удовлетворительным.

**6. Сортировка занимает много времени**

Решение: отредактировать сортировочные поля, отметить неиспользуемые. Использовать только сортировки-цепочки. Не использовать сортировки по реляционному пути.

в EME БД базовая версия EME.WMS имеет эталонную структуру БД с оптимально настроенными сортировками.

### Печать стека потоков на HTTP-сервере

Простой и эффективный способ обнаружения неоптимального кода. Подключиться к серверу WMS в момент максимальной нагрузки и выполнить печать стека через "ЕМЕ БД сервер менеджер" -> выбрать службу -> "Консоль сервера" -> вкладка "Пользователи" -> выбрать HTTP-сервер -> правой кнопкой мыши -> "Записать в журнал" -> "Стек вызова".

в системе EME.WMS печать стека потоков позволяет быстро обнаружить узкие места в обработке HTTP-запросов.
