# Записи и поля EME-L: создание, браузеры, цепочки, диалоги и работа с CEMERec

> **Синонимы для поиска:** запись EME БД, поле записи, структура базы данных, браузер, цепочка, один-ко-многим, ссылочное поле, CEMERec, реляционный путь, put_data, get_data, диалог, орган диалога, EME-L класс, конструктор класса, сортировка, SetSkipMode, SetChainMode, SetFindMode, SetRealMode, MustBeInRange, SetLinesLimit

---

## Создание новой записи и полей в ЕМЕ БД

Для создания новой записи в ЕМЕ БД необходимо перейти по пунктам меню "Система -> Ресурсы -> Структура Базы Данных". В появившемся диалоге нужно нажать на зелёный крестик или использовать сочетание клавиш `Ctrl - N`. После ввода имени новой записи изменения применяются сочетанием `Ctrl - S`.

Для создания нового поля записи нужно выбрать требуемую запись в диалоге "Структура Базы Данных" и нажать кнопку "Редактирование полей". Указывается название поля и его атрибут (тип данных). После выполнения действий изменения сохраняются сочетанием `Ctrl - S`.

в языке EME-L каждая запись представляет собой таблицу с построчной структурой. Поля записи могут иметь различные типы данных: строка текста, дата, время, ссылка и другие.

## Чтение и запись данных в ЕМЕ БД: put_data и get_data

ЕМЕ БД относится к системам класса RISC (системы с сокращённым набором команд). Используется пара команд для записи и чтения данных — `put_data()` и `get_data()`.

Обе команды обладают следующими параметрами:

- Номер Записи
- Номер Поля
- Физический номер строки
- Адрес блока данных в памяти
- Размер блока

Здесь "Номер Записи", "Номер Поля" и "Номер Строки" — целые числа, задающие адрес блока данных в БД. При использовании `put_data()` адрес содержит готовый блок данных для загрузки в БД. При использовании `get_data()` используется явная ссылка на адрес свободного буфера памяти для выгрузки данных из БД.

Благодаря данному механизму запись и чтение данных выполняется очень быстро и напрямую, без операций поиска и сравнения ключей.

в системе EME.WMS механизм `put_data`/`get_data` является базовым для всех операций с данными. Ядро БД автоматически формирует эти вызовы при работе с объектами CEMERec.

## Работа с браузерами в ЕМЕ БД

Работа с браузером — частая составляющая работы с ЕМЕ БД. Если на диалоге "Описание записи" нажать иконку тулбара "Листать записи" или горячую клавишу F3 ("Открыть браузер"), открывается окно браузера.

Каждая строка в браузере — это строка записи. Каждая колонка — поле записи. Порядок следования строк задаётся сортировками, порядок колонок задаётся при конструировании браузера программистом и может быть изменён перетаскиванием мышкой. Помимо колонок, привязанных к полям записи, могут быть колонки, отвязанные от записи — для их заполнения применяется специальная функция на EME-L или на C++.

Главный браузер диалога обычно показывает главную запись диалога. Другие браузеры применяются для заполнения полей ссылочными данными или вызываются специальными кнопками.

в языке EME-L браузеры являются основным средством отображения табличных данных пользователю.

## Создание связи "Один-ко-многим" (цепочки) в ЕМЕ БД

В ЕМЕ БД отсутствуют индексы и первичные/внешние ключи классических СУБД. Связи между записями реализуются через "ссылочные поля" (ссылки), а совокупность связанных записей называется "цепочкой". Это аналог связи "один-ко-многим".

Ссылочные поля помечаются символом `@`.

### Пример создания связи "Один-ко-многим"

Имеются записи "Клиенты" и "Звонки клиентов". Для связи "один-ко-многим" (у одного клиента несколько звонков):

**Запись "Клиенты":**

1. Создать запись
2. Создать поле "Фамилия" с типом "Строка текста"
3. Создать поле `@Звонки клиентов` с типом "Ссылка на начало цепочки". Во вкладке "Запись, на которую ссылается поле" указать запись "Звонки клиентов". Во вкладке "Поле-ссылка на заголовок" указать поле `@Клиент` из записи "Звонки клиентов"

**Запись "Звонки клиентов":**

1. Создать запись
2. Создать поле "Дата" с типом "Дата"
3. Создать поле "Время" с типом "Время"
4. Создать поле `@Клиент` с типом "Ссылка на заголовок цепочки". Во вкладке "Запись, на которую ссылается поле" указать запись "Клиенты". Во вкладке "Поле-ссылка на начало цепочки" указать поле `@Звонки клиентов` из записи "Клиенты"
5. Создать поле "Сортировка по клиенту" с типом "Сортировка-цепочка"
6. На вкладке "Поля записи: сортировка" первым полем указать `@Клиент` (всегда первым — ссылка на заголовок), вторым — "Дата"

После выполнения действий необходимо пройти тест целостности БД через "Система -> Тест целостности базы данных".

в системе EME.WMS цепочки являются ключевым механизмом связи данных. Корректная настройка цепочек критична для производительности.

## Создание EME-L класса в ЕМЕ БД

EME-L класс создаётся через редактор классов: "Система -> Ресурсы -> Редактор классов". В диалоге нажимается зелёный крестик, вводится имя нового класса, изменения сохраняются `Ctrl - S`.

Реализация конструктора пишется на вкладке "Конструктор". Методы класса — на вкладке "Методы". Наследуемый класс создаётся аналогично, с указанием родительского класса в атрибуте "Parent".

```EME-L
'Пример конструктора класса'
Constructor()
{
    m_r_Document = Object("dsDB", "Document");
    m_DBWatch = Object("DBWatch");
}
```

в языке EME-L классы являются основным средством организации бизнес-логики. Каждый класс может содержать конструктор, методы и переменные-члены.

## Создание диалога в ЕМЕ БД

Диалоги создаются через "Система -> Ресурсы -> Редактор диалогов". Открывается дефолтный шаблон, на основе которого создаются новые диалоги.

Для редактирования базовых свойств нужно дважды нажать на заголовок шаблона. Заполняются параметры:

- Софтовое имя диалога
- Заголовок диалога
- Базовая запись диалога

Если диалог нужно привязать к записи БД — снимается признак "Не привязывать диалог к записи БД".

### Размещение органов на диалоге

Размещение осуществляется с помощью панели инструментов над шаблоном:

- Кнопка `ab` — текстовое поле
- Кнопка `Aa` — статический текст
- Кнопка `XYZ` — орган группировки

Диалог сохраняется `Ctrl - S`.

### Привязка диалога к EME-L классу и записям

Для привязки диалога к EME-L классу нужно открыть свойства диалога, нажать `<= EMEL` и выбрать класс из списка. Для привязки к записи БД заполняется "Базовая запись диалога".

Органы диалога привязываются аналогично:

1. Задать имя органа (содержащее имя привязанного класса)
2. Выставить галочки "Привязать к полю БД" и "Тип поля взять из БД"
3. На вкладке "Привязка" указать поле записи БД

в системе EME.WMS корректная привязка диалогов к классам и записям БД является обязательным условием их работоспособности.

### Добавление нового пункта меню

Для добавления диалога в меню EME.WMS нужно в редакторе классов выбрать класс `Menu` и добавить строку в функцию `UpdateMenu(Line)`:

```EME-L
db.AddDialog("Справочники.&Покупатели", "ClientsHandbook.Dialog");
```

Первый параметр — путь до кнопки меню (амперсанд указывает горячую клавишу). Затем выполнить `UpdateUsersMenu` (клавиша F5) и перерегистрироваться.

## Сортировка данных в ЕМЕ БД

Для сортированного отображения данных в запись добавляется поле типа "Поле-сортировка". На вкладке "Поле записи: сортировка" указываются поля, по которым выполняется сортировка.

Поле типа "Поле-сортировка" — двунаправленный список. Ядро ЕМЕ БД поддерживает первую и последнюю строку в сортировке, а также быстрый поиск строк.

### Отсортированный браузер диалога

Для добавления отсортированного браузера в редакторе диалогов заходим в свойства диалога -> "Броузеры диалога" -> "Добавить новый". Выбираем базовую запись, поле сортировки и редактируем свойства. Затем привязываем браузер как главный через кнопку "Главный броузер".

## Чтение по реляционному пути в классах, производных от CEMERec

Чтение по реляционному пути — механизм получения связанных объектов через методы класса записи.

```EME-L
r_Document = Object("dsDB", "Document");
r_Document.SetLine(90);
code = r_Document.GetGates().GetCode();
```

Объект класса `Document` создаётся оператором `Object` и удаляется при выходе из метода. При первом вызове `GetGates()` объект создаётся и сохраняется в специальном массиве. При последующих вызовах возвращается тот же объект. Это позволяет использовать чтение по реляционному пути в цикле без размножения объектов.

### Корректная работа с реляционным путём

```EME-L
Loop(r_Document)
{
    'GetGates() возвращает один и тот же объект,
    'но он каждый раз перенастраивается на нужную строку'
    gatesCode = r_Document.GetGates().GetCode();
}
```

### Некорректная работа с реляционным путём

```EME-L
r_Document.SetFirstLine();
r_Gates1 = r_Document.GetGates();
r_Document.SetNextLine();
r_Gates2 = r_Document.GetGates();
'Ошибка: r_Gates1 и r_Gates2 — один и тот же объект!'
code1 = r_Gates1.GetCode();
code2 = r_Gates2.GetCode();
'code1 и code2 — код ворот второго документа'
```

в языке EME-L объекты, полученные по реляционному пути, не нужно проверять на `IsValidLine()`. Но объект записи, настроенный методами `SetLine()`, `SetFirstLine()`, `SetNextLine()`, всегда нужно проверять на валидность перед чтением.

## Режимы обхода записей CEMERec

При работе с записями часто возникает проблема быстродействия при росте данных. У класса `CEMERec` есть следующие режимы обхода:

- `SetRealMode()` — обход по физике, полный перебор без условий
- `SetSkipMode()` — перебор с наложением условий
- `SetSkipMode("Поле индекса")` — поиск по полю индекса
- `SetChainMode()` — обход по цепочке (самый быстрый)
- `SetFindMode()` — поиск по хэш-таблице

Установка любого режима отменяет все ранее установленные правила обхода.

```EME-L
r_DocLines = r_Document.GetDocLines();
r_DocLines.SetSkipMode();
r_DocLines.GetGoodsItemRefFld().MustBeRefEQ(goodsREf);
r_Registers.SetSkipMode(); 'Фильтр по ссылке удалён!'
```

в EME БД правильный выбор режима обхода критически важен для производительности на больших объёмах данных.

### SetSkipMode

Режим перебора с условиями. Работает относительно быстро благодаря блочному чтению. Хорошо использовать в скип-функциях браузеров или для поиска строки по идентификатору. Нельзя использовать в цикле.

```EME-L
r_Lines.SetSkipMode();
r_Lines.GetNameFld().MustBeEQ(name);
nStartLine = r_Lines.GetNoOfLines() - 4000000;
If(startLine > 0)
    r_Lines.SetLine(startLine);
    r_Lines.SetLinesLimit(2000000);
    r_Lines.SetNextLine();
Else
    r_Lines.SetFirstLine();
End If
```

### SetSkipMode("Поле индекса")

Режим поиска по полю индекса. Работает быстро благодаря использованию поля индекса. Допускается использование внутри цикла.

```EME-L
r_Stamps.SetSkipMode("Индекс номер марки");
r_Stamps.GetNumberFld().MustBeEQ(csStamp);
r_Stamps.SetFirstLine();
```

```EME-L
r_Stamps.SetSkipMode("Индекс партия - номер");
r_Stamps.MustBeIndexEQ(LotRef, SerialNo);
r_Stamps.SetFirstLine();
```

### SetChainMode

Самый быстрый способ обхода. Перебираются только строки цепочки, нет обхода по всей записи. При росте данных скорость не падает. Исключение — длинные цепочки, размер которых сопоставим с размером записи.

в EME БД цепочечный режим обязателен при работе со строками документов и заказов.

### SetFindMode

Режим поиска по хэш-таблице. Целесообразен при отсутствии индекса и необходимости найти в цикле не менее 10 значений. При первом поиске считываются значения полей для всех строк, вычисляется хэш и запоминается в памяти. Не рекомендуется вызывать в цикле — правильно один раз перед циклом.

### Влияние удалённых строк

В режиме `SetRealMode()` флаги читаются у всех удалённых строк. В `SetSkipMode()` тоже читаются флаги удалённых строк, начиная с первого условия. Если большинство строк удалено — первое условие должно отбрасывать удалённые строки (в удалённых строках все поля содержат нули, ссылочные — `NULL_REF`).

в языке EME-L `MustBeRefEQ()` по валидной ссылке отбросит удалённые строки, а по `NULL_REF` — пропустит.

## Лучшие практики оптимизации EME-L кода

1. Избегать дублирования кода. Не вызывать один и тот же метод многократно.

```EME-L
'Плохо — три вызова GetBoxMU():'
If (r_GoodsItem.GetBoxMU().IsValidLine())
    AddOutput("MUBox", r_GoodsItem.GetBoxMU().GetName());
    AddOutput("BoxFactor", r_GoodsItem.GetBoxMU().GetBaseFactor());
End If
```

2. Использовать сортировку-цепочку вместо `SetSkipMode()` когда возможно.

3. Ограничивать данные на всех этапах разработки.

4. Выделять уязвимые участки кода и оценивать риски работы с миллионами строк.

5. Избегать работы с большими данными на работающем складе. Использовать "мусорный документ" вместо удаления строк.

6. Тестировать программу и проверять производительность на реальных данных.

в системе EME.WMS оптимизация кода при работе с большими записями является критически важной задачей, влияющей на работу всего склада.
