## Обновление списка приказов на ТСД с HTTP-сервера

ТСД работает со списком приказов, загруженных в память с сервера. Фоновый поток на ТСД периодически отправляет запросы на сервер для обновления списка приказов.

Обновление происходит либо по таймеру каждые 10 секунд, либо после действий пользователя (нажатие кнопки "Обновить", авторизация и т.д.).

HTTP-запросы поступают в ядро, и оно решает, какой обработчик выполнять — код на C++ или обработчик на EME-L. Все запросы от ТСД идут на виртуальный каталог `terminals.html`, используется класс `Terminals`.

### Пошаговый механизм обновления списка приказов

1. В методе `CThreadData::UpdateData()` осуществляется отправка запроса с ТСД на сервер (формируется очередь запросов и параллельная отправка на транзакционный и нетранзакционный порты).

2. Перед формированием списка приказов выполняется проверка регистрации ТСД в системе и авторизация кладовщика методами `Terminals::CheckTerminal()` и `CommonTerminal::CheckOperator()`.

3. При успешных проверках вызывается `DrawListOrders::DrawListOrdersEx()`, формирующий список значений общих настроек системы. Далее вызываются методы в зависимости от типа приказов на ТСД.

4. Для каждого приказа формируется блок с параметрами (тип, номер документа и т.д.). Начало блока — тег `Order=Start`, окончание — `Order=End`.

5. После обработки сервером ответа вызывается `CThreadData::OnRequestDone()`, которому передаются данные в объекте `CHttpData`.

6. Данные из локального объекта сохраняются в глобальный `EMEGlobalData` методом `EMEGlobalData::SaveServerData()`. При изменении данных посылается сигнал `emit ServerDataChanged()`, вызывающий обновление экрана ТСД.

7. Метод `CThreadData::OnRequestDone()` анализирует необходимость немедленной отправки следующего запроса. Если во время обработки выполнено действие на ТСД — новый запрос отправляется мгновенно. За это отвечает `CHttpData::IsSendRequestImmediately()`.

в системе EME.WMS обновление списка приказов является ключевой операцией, от скорости которой зависит эффективность работы кладовщиков.

## Оптимизация обмена данными между ТСД и сервером

Каждые 10 секунд ТСД отправляет запрос `GetListOrders` на сервер. Сервер отвечает командой `TakeListOrders` со списком приказов.

Сервер не каждый раз присылает полную информацию — может прислать "урезанные" данные (только номера строк и изменяемую информацию). Полные данные запрашиваются при обнаруженных расхождениях.

### Добавление изменяемого поля в оптимизацию

Для добавления поля, которое может меняться одновременно на ТСД и сервере:

1. В EME-L классе `DrawListOrders` требуемое поле добавляется в запрос методом `AddOutputKeys`.

2. В методе `IsSameOrders()` (класс `EMEGlobalData`) добавляется сравнение для требуемого поля.

### Механизм оптимизации обмена

1. ТСД определяет, нужны ли полные данные, через `EMEGlobalData::IsGetOnlyOrdersKeys()`. Если полные данные не нужны — передаётся признак `GetListOrdersKeys`.

2. Сервер формирует ответ с учётом `GetListOrdersKeys`.

3. В EME-L классе `DrawListOrders` устанавливается признак `m_bGetListOrdersKeys`.

4. ТСД получает ответ и анализирует признак `GetListOrdersKeys`.

5. `EMEGlobalData::IsSameOrders()` сравнивает ответ с локальным списком.

6. При расхождениях с установленным `GetListOrdersKeys` незамедлительно отправляется запрос на полные данные через `CHttpData::IsSendRequestImmediately()`.

в EME БД оптимизация обмена данными критически важна на складах с большим количеством ТСД, где нагрузка на сервер может достигать сотен тысяч запросов в сутки.

## Пакетная обработка HTTP-запросов от ТСД сервером заданий

На больших складах ежесуточно выполняется более 150 000 серверных транзакций, инициируемых с ТСД. Для уменьшения нагрузки несколько запросов с разных ТСД объединяются в один с общей транзакцией. Обработка перенесена из потоков HTTP-сервера в поток сервера заданий.

### Формирование HTTP-запросов со стороны ТСД

- `CHttpData::GetRequestString()` — формирует строку запроса. Запросы делятся на транзакционные и нетранзакционные.

- `CEMEHttp::HttpRequest` — формирует порт, URL и отправляет запрос. Сервер не выполняет запрос сразу — возвращает ID запроса, ТСД запоминает его в `ServerJobNo`. Используется для цикличной отправки запроса статуса задания.

- `CThreadData::OnRequestDone` — формирует интервал между отправками запросов статусов.

### Обработка HTTP-запросов на стороне сервера в EME-L

- `Document_OnBatchRespond(Doc As "HTMLDocument")` — сервер заданий получает наименование класса по команде.

- `Document_OnBatchRespond(arrHtmlDocs As "Array", arrEMELObjects As "Array", ErrArray As "Array")` — обработка пачки запросов.

- `ParseCommand()` — определяет EME-L класс-обработчик по наименованию команды.

- `CheckTerminalBatch()` — пакетный поиск терминалов в БД и проверка версии программы ТСД.

в системе EME.WMS пакетная обработка HTTP-запросов обеспечивает масштабируемость системы на крупных складских комплексах.

## Настройки ТСД: флаги и параметры диалогов

В системе EME.WMS диалоги, работающие на ТСД, имеют ряд настраиваемых флагов:

- **Запрет дублирования диалогов** — запрещает одновременное открытие двух диалогов.
- **Не выводить панель инструментов** — скрывает верхнюю панель инструментов.
- **Создавать новую строку при старте** — диалог открывается с пустыми значениями.
- **Проверять отпечаток** — настройка не используется в текущей версии.
- **Защищать информацию от одновременного доступа серверным семафором** — активирует механизм защиты одновременного доступа к общим ресурсам.

в EME БД настройки диалогов ТСД конфигурируются через редактор диалогов и влияют на поведение интерфейса терминала.
