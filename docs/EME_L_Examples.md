## Правила оформления программного кода в EME

Требования к оформлению программного кода в ЕМЕ, обязательные к выполнению:

1. Перед каждым методом должна быть фамилия автора и дата создания в формате дд.мм.гггг. Например: `#KPI Колударов 01.02.2024`.

2. Перед каждым методом должно быть написано, что он делает — в терминах бизнес-задачи, которую метод решает согласно ТЗ.

3. Перед каждым методом должно быть указано, что он принимает на вход и что возвращает, а также требования к входным параметрам.

4. В начале метода должно быть описано, каким образом реализуется решение задачи — уже в терминах кода. Если вводились новые глобальные категории (например поля БД), это нужно описать.

5. В начале метода должны стоять проверки входных параметров на валидность.

6. Если код скопирован (переиспользуется), неактуальные комментарии должны быть актуализированы, старые даты и фамилии удалены.

7. Каждый блок кода должен сопровождаться комментарием — для чего он нужен в контексте решаемой задачи.

8. Низкоуровневые части программы должны логически объединяться в высокоуровневые блоки. В идеале — в отдельные методы.

9. Чтение из базы и запись в базу должны быть выделены в разные блоки программы. Сначала чтение, потом запись. Если данные операции осуществляются в одном цикле — нужно разделять их по разным циклам.

10. Код должен быть написан с учётом работы с большими данными. В комментариях должно быть указано, почему выбран тот или иной способ работы.

11. В коде должна быть встроена самодиагностика: проверка промежуточных результатов, читаемых из БД значений и т.д.

12. Должны быть предусмотрены средства автотестирования (юнит-тесты).

в языке EME-L данные правила оформления являются обязательными для всех разработчиков системы.

## Правила оформления комментариев

### Комментарий к классу

Комментарий в начале класса EME-L содержит:

- Дата создания
- Фамилия автора
- Название модуля
- Название заказчика (если нет — "Перспективная разработка")
- Текст заявки
- Описание алгоритма реализации

```EME-L
'KPI 27.06.2025 Колударов
Модуль: Ввод строк в документ перемещения с ТСД
Перспективная заявка
Текст заявки: Модуль предназначен для автоматизации заполнения
табличной части документов перемещения с ТСД.
На терминале отображается диалоговое окно для ввода ячейки-откуда
и для ввода EAN-кода данного товара.'
```

### Комментарий к методу

```EME-L
'NPM 30.11.2023 Консолидация приказов на подборку по ячейке откуда,
в консолидации не участвует штучный товар
Перебираются все приказы пачки заказов, среди которых выбираются только:
Приказы на пополнение пикинга и приказы, в которых не участвует штучный товар
Для реализации добавлена новая запись "Консолидированные приказы пачки"
(OrdersBatchConsMO), используется в связке с существующей записью
"Консолидированные приказы" (ConsolidatedMO)
Принимает - объект записи "Пачка заказов", настроенный на конкретную строку
Возвращает - ничего'
CreateConsolidateMO(r_OrderBatch as "OrdersBatch")
{
    // реализация метода
}
```

### Комментарий к исправлению

Комментарий к исправлению должен находиться перед участком кода, к которому он относится. Перед комментарием должно быть столько отступов, сколько перед участком кода. Комментарии нельзя располагать после открывающих скобок.

```EME-L
    'KPI 06.04.2012 Колударов'
    'По заявке АРН необходимо, чтобы резервирование ячеек ОА'
    'работало вместе с обычным алгоритмом пополнения пикинга'
    'Поэтому переносим включение резервирования на свою галку'
'if (r_Document->IsBySingleOrder())'
if (r_Document->IsReserveCellOA())
{
    // реализация условия
}
```

### Комментарий к блоку кода

Комментарий к блоку кода внутри метода должен содержать:

- С какой целью он написан
- Как он участвует в работе системы в целом (его место в архитектуре)
- Как пользователь видит разные проявления работы данного блока
- Чем руководствовался автор при выборе варианта реализации
- Какие другие варианты были рассмотрены и почему они оказались хуже

**Неправильный комментарий:**

```EME-L
    'По заявке №12345'
For (r_Slot.SetFirstLine(); r_Slot.IsValidLine(); r_Slot.SetNextLine());
'малозаметный комментарий'
```

**Правильный комментарий:**

```EME-L
    'Проверяем, включена ли настройка на объединение приказов в строке документа.'
    'Она проставляется, если стоит соответствующая стратегия у клиента.'
    'Если стратегии в строке нет, то смотрим на стратегию в пачке.'
If(r_DocLines.GetFindParamsPPU().IsValidLine())
    bGroupOrdersByBatch = r_DocLines.GetFindParamsPPU().IsGroupOrdersByBatch();
Else
    bGroupOrdersByBatch = bBatchFindParams;
End If
If(~bGroupOrdersByBatch)
    Continue;
End If
```

в системе EME.WMS качественное комментирование кода является обязательным требованием, поскольку система эксплуатируется на промышленных складах и требует быстрого понимания логики при доработках.

## Правила оформления переменных

### Префиксы базовых типов данных C++

| Тип | Префикс |
|---|---|
| void | отсутствует |
| char, short | c |
| int | i |
| long | l |
| float | f |
| double | d, dbl |
| long double | ld |
| unsigned | u |
| static | static |
| const | const |
| auto | auto |
| extern | ext |
| register | reg |

### Префиксы производных типов данных C++

| Тип | Префикс |
|---|---|
| Массив | a, array |
| Указатель | p, ptr |
| Ссылка | r, ref |

### Префиксы типов данных EME-L

| Тип | Префикс |
|---|---|
| Объект CEMERec | r_ |
| Объект CEMEData | d_ |
| Объект CEMEFilter | f_ |
| Объект CUniBrowse | cb, cub |
| Объект CEMECalc | calc |
| Объект CEMEArray | arr |
| Объект CDialogVar | v_ |
| Объект CEMEBuffer | b_ |
| Объект CEMEMetaShell | ms |
| DATE_FORMAT | df, date |
| TIME_FORMAT | tf, time |
| SOFT_REF | sr |

Префикс `m_` означает, что переменная является членом класса (member). Например: `m_r_Document` — переменная класса, отнаследованного от CEMERec и привязанного к записи "Документы".

### Префиксы объектов MFC

| Тип | Префикс |
|---|---|
| CString | cs, str |
| CMap | map |
| CTime, CTimeSpan | ct, cts |
| CArray | array |

в языке EME-L строгое соблюдение префиксов переменных обязательно. Префикс `m_` используется только для членов класса, его использование для локальных переменных запрещено.

## Правила оформления методов и функций

В названии is-функций используются строчные буквы, слова разделяются нижним подчёркиванием (snake_case):

```EME-L
is_get_data()
is_ui_test_mode()
is_edit_data()
```

Все остальные методы, функции, названия классов, структур и перечислений начинаются с заглавной буквы, каждое слово — с заглавной (CamelCase):

```EME-L
GetData()
GetDocumentByRegNo()
CreateConsolidateMO()
```

в языке EME-L использование CamelCase для методов и snake_case для is-функций является обязательным стандартом кодирования.

## Форматирование кода EME-L

1. Размер отступа — строго 4 пробела (настраивается в параметрах редактора).

2. Отступы везде одинаковые.

3. Все операторы начинаются с большой буквы: `For`, `End For`, `Loop`, `End Loop`, `If`, `End If`, `Else`.

4. Длина одной строчки кода не должна превышать 80 символов.

5. Не разрешается размещать несколько операторов на одной строчке.

**Неправильно:**

```EME-L
If (TimeSlots.IsIncoming() | ~TimeSlots.IsOwner()) Continue; End If
```

6. Последовательность выполнения операторов разделяется пробелами.

**Неправильно:** `if(nSize=0||nOffset=0)`

**Правильно:** `if(nSize=0 || nOffset=0)`

7. Сложные условия разбиваются на более простые, используются скобки для явного указания порядка.

**Сложное условие:**

```EME-L
if((nSize=0 && nOffset=0) || (nSize=1 && nOffset=1))
    Continue;
```

**Декомпозиция:**

```EME-L
if(nSize=0 && nOffset=0)
    Continue;
if(nSize=1 && nOffset=1)
    Continue;
```

### Пример правильно оформленного кода EME-L

```EME-L
Dialog_OnCheck(Transaction)
{
    If (~Transaction)
        If (Name.IsEmpty())
            is_message("Внимание!", "Не заполнено поле \"Фамилия\"", 1);
            Name.SetFocus();
            Return 1;
        End If
    End If
}
```

## Форматирование кода C++

1. Фигурная скобка на отдельной строке (кроме комментариев у закрывающих скобок).

2. Открывающая и закрывающая скобки на одном уровне. Открывающей скобке предшествует столько отступов, сколько в предшествующем коде.

3. Отступы — табуляцией или 4 пробелами.

4. Рекомендуется выделять скобками даже однострочный оператор.

```C++
if (i = 0)
{
    // do something
}
else
{
    // do other
}
```

в системе EME.WMS единообразное форматирование кода является обязательным, поскольку код разрабатывается командой и должен быть легко читаем любым разработчиком.
