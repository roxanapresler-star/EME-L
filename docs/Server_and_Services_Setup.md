## Создание сервера EME.WMS

Создание сервера EME.WMS осуществляется при помощи утилиты `service.exe`. Для корректной работы сервера WMS необходимо создать и запустить **три отдельные службы**:

- **Служба сервера** (ключ `/server`)
- **Служба мозгового компьютера** (ключ `/bc`)
- **Служба http-сервера** (ключ `/http`)

Для каждой службы нужно подготовить **отдельные директории с данными** и произвести настройку IP-адреса и порта. Порты служб **не должны повторяться**.

в системе EME.WMS первоначально служба запускается на локальном IP. Чтобы настроить адрес сервера транзакций, нужно остановить службу, открыть рабочую станцию в моно-режиме, перейти в настройки рабочей станции (`Ctrl + Shift + W`), выбрать вкладку **«Сеть»** и указать IP-адрес компьютера в поле **«IP адрес сервера»**. Порт сервера настраивается в поле **«Порт прослушивания»**. Аналогичная операция выполняется для IP-адреса и порта HTTP-сервера на вкладке **«HTTP-сервер»**.

### Создание файла параметров службы

Далее необходимо запустить `db.exe` от имени администратора и создать файл параметров службы. Если имеется настроенный файл `eme.config`, нужно при выборе конфигурации EME.WMS нажать **«Добавить»**, затем выбрать **«Выбрать папку с данными и параметры запуска»**. В новом окне выставляются параметры:

| Параметр | Значение |
|----------|----------|
| Сетевой режим — сервер | Для службы сервера |
| Сетевой режим — рабочая станция, тип — `/fullvm` | Для службы http-сервера и мозгового компьютера |
| Сервисы — `/bc` | Только для службы мозгового компьютера |
| Сервисы — `/http` | Только для службы http-сервера |
| Заголовок БД | Путь к `.hdr` файлу — **разный для всех трёх служб** |
| Наименование службы | Название службы |
| Путь к файлу параметров | Путь к корневой папке WMS |

После заполнения параметров нажать **«Установить службу»**.

### Запуск службы через server manager

Созданный файл параметров службы появится в корневой папке WMS. Внутри файла находятся ключи запуска `/service` и `/meta` с параметрами, а также дополнительные ключи запуска.

в системе EME.WMS для запуска службы нужно перейти в директорию `kernel.x64/exe` и запустить утилиту `service.exe`. Откроется диалог **«ЕМЕ БД сервер менеджер»**. Через этот диалог можно:

- Запускать/останавливать службы
- Выполнять утилиты сервера
- Просматривать консоль сервера и `CATASTRF.log`

Для запуска службы нажать **«Запустить»**. В `CATASTRF.log` должны отобразиться сообщения:

- `Сервер транзакций загружен. Интерфейс (IP:порт)` — для службы сервера и мозгового компьютера
- `WEB-сервер транзакционных запросов загружен. Интерфейс (IP:порт)` — для службы http-сервера

Если служба запускается и тут же останавливается — нужно запустить рабочую станцию в моно-режиме, выйти из неё и повторить запуск. Статус службы должен измениться на **«Работает»**.

в EME БД консоль сервера содержит вкладки:

- **Пользователи** — подключённые компьютеры и пользователи с данными по отправке/получению
- **Семафоры**
- **Настройки** — изменение настроек рабочей станции для сервера
- **Журнал работы** — вывод информации из `CATASTRF.log`
- **Транзакции** — статистика: количество, продолжительность, время ожидания, средняя длительность

## Просмотр серверных транзакций в ЕМЕ БД

Диалог **«Просмотр серверных транзакций»** доступен по пунктам меню **«Система → Ресурсы → Просмотр серверных транзакций»**, а также через любой диалог создания электронных документов (Приход, Отгрузка, Списание и др.) — правой кнопкой мыши по документу → **«Просмотр серверных транзакций»**.

в системе EME.WMS просмотр серверных транзакций возможен **только в сетевом режиме**. В моно-режиме данные о транзакциях отсутствуют. Заготовленные данные можно скачать по адресу `\\srv002\db_box_5.0\sys\dat_server` — три директории: `transactions-local`, `transactions-server` и `archives`. Их нужно поместить в папку с данными на сервере WMS.

### Параметры поиска транзакций

Для поиска информации указываются:

- Название записи
- Номер строки
- Номер записи
- Период поиска

Номера строк и записи определяются из диалога создания документов (кнопка «+»). Данные автоматически заполняются, если открыть просмотр транзакций из диалога документов.

в языке EME-L после нажатия **«Показать»** выводятся данные по транзакциям. Ключевые поля таблицы:

| Поле | Описание |
|------|----------|
| Номер | Идентификатор транзакции для поиска в логах |
| Дата / Время | Момент выполнения |
| Транзакция | Название транзакции |
| Данные | Содержимое транзакции |
| Запись | Запись в БД |
| Поле | Поле БД; значение с `@` — ссылка на запись |
| IP / Компьютер / Оператор | Источник действия: оператор, сервер или ТСД |

## Организация файлов и директорий программы EME.WMS

в системе EME.WMS корректная организация файловой структуры критична для продуктивной среды. Корневая директория — **`EME.WMS`**.

Для каждого компонента EME.WMS (рабочая станция, сервер БД, http-сервер, мозговой компьютер) создаются поддиректории:

- `dat_{название компонента}` — директория с данными
- `kernel.x64` — исходники ядра
- `metaproj` — исходники метапроекта

В каждой поддиректории должны находиться `lnk`-файлы (ярлыки), согласованные с ключами запуска: `mono.lnk`, `server.lnk`, `mono-http-bc.lnk`.

Дополнительно:
- Для http-сервера — директория `web/` (обновление ТСД по сети)
- Для мозгового компьютера — директория `etc/` (необходимое для работы)

### Пример структуры каталога EME.WMS

```
EME.WMS/
├── workstation/
│   ├── dat_workstation/
│   ├── kernel.x64/
│   ├── metaproj/
│   ├── mono.lnk
│   └── ws.lnk
├── server/
│   ├── dat_server/
│   ├── kernel.x64/
│   ├── metaproj/
│   ├── mono.lnk
│   └── server.lnk
├── http/
│   ├── dat_http/
│   ├── kernel.x64/
│   ├── metaproj/
│   ├── web/
│   ├── http.lnk
│   └── mono.lnk
└── bc/
    ├── dat_bc/
    ├── etc/
    ├── kernel.x64/
    ├── metaproj/
    ├── mono.lnk
    └── bc.lnk
```

## Перенос транзакций на сервер заданий

в системе EME.WMS перенос ресурсоёмких транзакций на сервер заданий — способ оптимизации. Ресурсоёмкие транзакции выполняются дольше обычного, особенно на слабых компьютерах, и мешают работе всего склада.

**Задание** — это специальный EME-L класс. В диалоге **«Классификатор Web объектов»** должна быть настроена точка входа `ServerJob`.

Пример ресурсоёмкой транзакции — просчёт пачки заказов в классе `ServerJobOrdersBatch`. У этого класса есть серверная и клиентская части.

### Конструктор класса задания

```EME-L
Document = Object("dsDB", "OrdersBatch");

m_FullFileNameDump;
m_TraceObj As "Trace";

strMess AsExternal;
Report AsExternal As "Report";
NoOfLines AsExternal;
```

в языке EME-L переменные, объявленные как `AsExternal`, позволяют заданию вернуть значение на рабочую станцию. Возвращаемые значения: `strMess` (результат просчёта), `Report` (отчёт «Dump просчёта пачки»), `NoOfLines` (количество непросчитанных строк).

### Серверная часть задания

в системе EME.WMS серверная часть любого класса, связанного с `ServerJob`, содержит исполнительный метод (в `ServerJobOrdersBatch` — метод `Run()`). Результат выполнения задания — значение, возвращаемое исполнительным методом. Дополнительно имеются вспомогательные методы: `WarningCreateFile`, `GetFullFileNameDump`, `GetTraceObjectBatch`.

### Клиентская часть задания

Для обращения к серверу заданий используется C++ класс **`ServerJob`**. Главный метод — `SendJob`, отправляющий задание на сервер:

- Параметр — имя задания
- Возвращаемое значение — уникальный идентификатор задания (целое число)

Задание выполняется **асинхронно** через очередь заданий. Имя задания должно быть понятным. В классе `ServerJobOrdersBatch` имя формируется методом `GetJobName`.

в языке EME-L параметр `"TRANSACTION"` в методе `SendJob()` указывает, что при выполнении задания будет открываться транзакция. Параметр `"REPORT"` — транзакция не открывается. У сервера заданий **две очереди**: транзакционная и нетранзакционная, использующие два разных порта — это исключает блокировку заданий.

Выполненное задание хранится на сервере заданное время (по умолчанию 1 час). Для получения возвращаемого значения используется метод `GetJobReturnValue` класса `ServerJob`.

### Пример доработки EME-L класса для сервера заданий

Добавить в конструктор переменную `ActiveJobId` со значением `Empty`. Определить её в методе `DialogOnAfterUpdateFunction()`. В коде класса найти блок просчёта пачки и добавить условие:

```EME-L
If (m_bNoAskMakePalletPickingUpForServer & is_workstation_mode()=="WORKSTATION" & is_message(MsgCaption, tr("Выполнить просчет пачки заказов на сервере?"),"YESNO","QUESTION") == "YES")
    ServerJob = Object("IClass", "ServerJobOrdersBatch", "ServerJobOrdersBatch");
    ActiveJobId = ServerJob.SendJob(Document.GetLine(), arrOrders, True);
```

### Обработка события завершения задания

в языке EME-L для получения сообщения от сервера о завершении задания используется метод `Dialog_OnEvent`:

```EME-L
Dialog_OnEvent(NotifyHdr As "EME_HDR_NOTIFY")
{
    Params As "Parameters" = NotifyHdr.GetNotifyParam();
    Id = Params.GetLongParam("Id");
    If (Id != ActiveJobId)
        Return;
    End If

    Status = Params.GetLongParam("Status");
    If (Status < 3)
        Return;
    End If

    ServerJob = Object("ServerJob");
    strMess = ServerJob.GetJobReturnValue(Id, "strMess");
    NoOfLines = ServerJob.GetJobReturnValue(Id, "NoOfLines");
    MsgHeader = tr("Просчет пачки!");
    strMess = strMess + tr("Показать результат поиска продуктов?");
    If(is_message(MsgHeader, strMess, "YESNO") == "YES")
        Report As "Report" = ServerJob.GetJobReturnValue(ActiveJobId, "Report", True);
        Report.ShowWindow();
        If(RunReportDump(Document.GetFullFileNameDump()) == 0)
            is_message(tr("Просчет пачки"),tr("На складе не найден ни один из продуктов пачки заказов."), 1, 3);
        End If
    End If

    is_ws_align_data();
    Dialog.Update();
    Object("Console").Hide();
}
```

### Метод OnOpenJob для журнала сервера заданий

Чтобы из журнала сервера заданий открывать саму пачку заказов (а не карточку задания), реализуется метод `OnOpenJob`:

```EME-L
OnOpenJob(Id)
{
    ServerJob = Object("ServerJob");
    OrdersBatchRef = ServerJob.GetJobArgValue(Id, 0);
    Object("Dialog", "OrdersBatch.Main.Dialog").ShowLine(True, OrdersBatchRef, 2);
    Return 1;
}
```

Если заданием является отчёт, а не транзакция — добавляется проверка:

```EME-L
OnOpenJob(Id)
{
    ServerJob = Object("ServerJob");
    If (ServerJob.GetJobStatus(Id) == 3)
        Report AS "Report" = ServerJob.GetJobReturnValue(Id, "Report", True);
        Report.ShowWindow();
        Return 1;
    End If
}
```

## Монитор Заданий

### Работа с Монитором Заданий

в системе EME.WMS Монитор Заданий является альтернативой рабочей станции для управления складом.

- Фронтэнд: `https://gitlab.eme.ru/metaproj-any/task-monitor/task-monitor`
- Ядро: `https://gitlab.eme.ru/metaproj-any/task-monitor/task-monitor-core`

Обмен данных с WMS реализован через JSON-файлы, которые парсятся в мониторе в ядерных классах (папка `network`) и в EME-L в классах `ProcessModule`.

### Подключение Монитора Заданий к EME.WMS

Подключение происходит через http-службу, по аналогии с ТСД. В настройках соединения указывается IP и порт http-сервера. В графе «Пользователь» указывается идентификатор оператора БД и пароль.

### Инициализация стартующих заданий

в языке EME-L если рабочая станция запущена в моно-режиме, бот стартующих заданий не отработает автоматически. Нужно вручную запустить метод `Test()` класса `AutoCreateTasksBot` (клавиша F5).

### Создание шаблонов заданий

Шаблон заданий — сущность, формирующая алгоритм работы задания и набор атрибутов.

**Пошаговая инструкция:**

1. Определить концепцию задания
2. Создать EME-L класс задания (ветка `ProcessModule.ProcessTaskModule.ProcessTasks`)
3. Сформировать шаблон через диалог **«Шаблоны заданий процесса»**
4. Сформировать параметры шаблона
5. Привязать шаблон к событию процесса
6. Назначить задание в список доступных оператору БД

**Типы заданий:**

- Стартующие (создаются ботом)
- Стандартные (реакция на событие)

**Параметры задания:**

- Поле ввода (обязательное/необязательное, множественный выбор)
- Кнопка (старт/завершение/промежуточное действие)
- Настройка (использует настройки WMS)

**Класс задания** называется по шаблону: `ТипДокумента-СутьЗадания` (например: `IncomeArrivalCar`, `OutcomeCreateBatch`, `InventoryCloseDocument`).

### Формирование шаблона задания

в системе EME.WMS шаблон формируется через диалог **«Шаблоны заданий процесса»** (меню: «Классификаторы → Бизнес-процессы → Шаблоны заданий»).

Ключевые параметры:

| Параметр | Описание |
|----------|----------|
| Задание | Название задания |
| Тип | Вечное / по событию |
| Класс | EME-L класс задания |
| Тип процесса | Бизнес-процесс |
| Тип терминала | Обычно `Virtual EME` |
| Пост-событие | Событие по результату выполнения |

Вкладки диалога: **Параметры** (ID, наименование, приоритет), **Обработчики** (методы), **Правила** (скрытие необязательных полей), **Разное** (описание и инструкция).

Ключевой метод обработчика — **`OnCommand`**. Команды: `TASK_COMMAND_EXECUTE_ID` (создание) и `TASK_COMMAND_CONFIRM_ID` (завершение). Результат — `ResultJSON` с 8 полями (рекомендуется менять только `Result` и `ResultText`).

### Формирование параметров шаблона

На вкладке «Параметры» создаётся новая строка с ID. Типы параметров: поле ввода, настройка, кнопка, команда, таблица. Способы ввода: текст + список и др. Основные методы: `_OnInput`, `_FillList`, `_GetList`.

### Привязка шаблона к событию и назначение оператору

в системе EME.WMS привязка осуществляется через диалог **«События процесса»** (новая запись → заполнить название, тип процесса, обработчик «Создать задание», событие, шаблон задания).

Затем задание добавляется в процесс через диалог **«Описание бизнес-процессов»** (процесс должен быть на статусе «Утвержден» → кнопка «Создать подпроцесс» → описание → событие).

Назначение оператору: диалог **«Складские должности»** → «Оператор БД» → вкладка «Задания» → добавить наименование задания с типом терминала `Virtual EME`.

### Настройка Монитора Заданий

**Пошаговая инструкция:**

1. Завести пользователя с должностью «Оператор БД» через **«Классификатор складского персонала»** (снять галку «Запрет автопоиска» на вкладке «Работа с ТСД»)
2. Экспортировать настройки процессов из системного филиала → импортировать в БД клиента (диалог «Описание бизнес-процессов» → «Дополнительно» → «Экспортировать»/«Импортировать»)
3. Включить пересылку событий: настройки модулей (`Ctrl+Shift+M`) → «Предпочтения → Процессы → Включить» = «ДА»; «Предпочтения.Процессы.Расширенный лог» = «ДА»
4. Настроить шаблоны заданий, параметры и события
5. Инициализировать контекстный поиск (меню: «Классификаторы → Бизнес-процессы → Параметры контекстного поиска») и проверить наличие терминала `Virtual EME`
6. Настроить бота стартующих заданий (статус «Утвержден», событие «BOTCT» с обработчиком «Создать задание»)
7. Проверить будильник ТМ01 (меню: «Утилиты БД → Навигатор канальной схемы → Редактор событий»): выполняется 1 раз в минуту, вызывает метод EME-L `TM01`
8. Настроить IP-адрес и порт HTTP-сервера в настройках рабочей станции (`Ctrl+Shift+W`)
9. Настроить автообновление: указать директорию данных в настройках «HTTP сервер → Директория данных»

### Структура директории для автообновления

в системе EME.WMS директория данных для автообновления Монитора Заданий:

```
{директория данных}/
└── monitor/
    ├── taskmonitor.update   (DESKTOP_VERSION и ANDROID_VERSION)
    └── update.zip
        ├── appupdater.jar
        └── update.jar
```

Если директория данных не указана, автоматически используется папка `web` в корне проекта EME.WMS.
